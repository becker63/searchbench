You are doing a very aggressive refactor of this repository.

Goal:
Make the state machine the single orchestration abstraction and delete as much transitional / compatibility code as possible without breaking the core behavior. We are no longer preserving legacy wrapper APIs just because old tests expect them. Favor deletion over adaptation.

Architectural target:
- loop_machine.py is the orchestration core.
- state-machine listeners own transition/iteration/repair span lifecycle.
- loop.py becomes a thin bootstrap / dependency wiring layer, not a second orchestration engine.
- tests should validate machine behavior and externally visible outcomes, not legacy wrapper compatibility.

Hard requirements:
1. Delete legacy compatibility surfaces aggressively.
2. Prefer removing code and rewriting tests over adding shims.
3. Do not preserve monkeypatch compatibility for deprecated functions.
4. Keep type annotations strong and explicit.
5. Keep behavior around accepted policy metadata, pipeline mutation, repair attempts, and scoring outcomes.
6. Do not invent new abstraction layers unless they clearly remove more code than they add.

Known high-value deletion targets:
- In loop.py, delete synthesize_valid_policy(...) entirely.
- Delete _ORIGINAL_SYNTHESIZE.
- Delete the run_loop branch that detects monkeypatched synthesize_valid_policy and adapts to the old tuple-based API.
- Delete repair_override / repair_via_synthesize compatibility plumbing unless there is a truly unavoidable internal use.
- Remove manual span ownership that duplicates listener ownership.
- Collapse wrapper-oriented tracing tests into a much smaller set of machine/listener tests.

Strong bias:
If a piece of code exists only to preserve an old API shape, old tuple return contract, old monkeypatch seam, or old tracing wrapper semantics, delete it.

Specific refactor instructions:

A. loop.py
- Make run_loop a thin entrypoint that:
  - builds dependencies/context,
  - instantiates the machine,
  - runs it,
  - converts the result into the existing public history/output shape only if needed.
- Remove any embedded “mini-orchestrator” logic that duplicates loop_machine.py.
- Remove the synthesize_valid_policy compatibility wrapper entirely.
- Remove old tuple-based repair result flow.
- Remove dynamic compatibility checks against old function identities.
- Inline or delete helpers that only existed to support the wrapper path.
- Keep only helpers that are true dependencies of the machine.

B. loop_machine.py
- Make listeners the sole owners of repair / iteration / transition span lifecycle.
- If RepairStateMachine.run() manually initializes span state that listeners already manage, remove that duplication.
- Keep listeners that carry real architectural value.
- If there is a transition tracing listener that is not materially used by tests or outputs, consider deleting it.
- Prefer a smaller, more explicit listener set over generic tracing scaffolding.
- Tighten machine model/context/result types where possible.

C. loop_types.py
- Remove or collapse types that only exist to bridge legacy wrapper outputs into machine outputs.
- Keep the public result model clear and strongly typed.
- Prefer one canonical outcome/result shape per machine.
- Remove redundant translation helpers if the machine result can be used directly.

D. tests
- Rewrite tests_harness/test_loop_failures.py so it no longer targets synthesize_valid_policy or monkeypatches it back into run_loop.
- Delete tests that only verify deprecated wrapper compatibility.
- Aggressively shrink tests_harness/test_tracing_wrappers.py; replace wrapper-plumbing assertions with a few focused listener/machine assertions.
- Keep and strengthen tests_harness/test_loop_machine.py as the main behavioral contract.
- Preserve tests for:
  - repair retries / exhaustion,
  - pipeline mutation resulting in accepted on-disk policy,
  - accepted_policy_source / changed_by_pipeline metadata,
  - failed candidate not being marked accepted,
  - score/metrics propagation,
  - flush/score hooks if still part of public behavior.
- If a test exists only because of the removed wrapper API, delete it rather than adapting production code to satisfy it.

E. observability
- Keep lower-level observability in runner.py / writer.py / pipeline if it is leaf-operation observability.
- Remove orchestration-level tracing from helpers/wrappers when the machine/listeners already own it.
- We want a clean split:
  - machine/listeners = orchestration observability
  - runner/writer/pipeline = leaf-operation observability

Implementation style:
- Prefer surgical deletions and collapsing code paths rather than moving old code around.
- Prefer direct calls and clear names over compatibility indirection.
- Remove dead comments/docstrings referring to the old wrapper architecture.
- Update LANGFUSE.md or any related docs if they still describe loop.py as the tracing/orchestration center.

Validation requirements:
- Run:
  - uv run ruff check
  - uv run basedpyright
  - uv run pytest
- If full pytest is too large during iteration, run targeted tests first, but finish with full pytest.
- Fix failures by deleting stale tests when they only preserve removed compatibility behavior.
- Do not reintroduce shims just to satisfy old tests.

Deliverables:
1. The refactor itself.
2. A short summary of:
   - what compatibility code was deleted,
   - which tests were removed vs rewritten,
   - any listeners added/removed,
   - net LOC removed (rough count is fine).

Success criteria:
- No synthesize_valid_policy compatibility wrapper remains.
- No _ORIGINAL_SYNTHESIZE remains.
- No run_loop monkeypatch compatibility branch remains.
- Span lifecycle for repair/iteration is listener-owned rather than split across listeners and imperative wrapper code.
- Tests primarily target machine behavior, not wrapper compatibility.
- The codebase is materially smaller and conceptually cleaner after the pass.

Be aggressive. Choose deletion whenever there is a tie.
